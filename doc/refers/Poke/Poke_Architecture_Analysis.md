# Poke 系统架构分析

## 1. 项目概述

Poke 是由 The Interaction Company of California（位于 Palo Alto）开发的 AI 助手聊天机器人，通过 iMessage/WhatsApp/SMS 与用户交互。该产品于 2025 年 9 月 8 日发布，因其独特的个性化交互和实用功能引起广泛关注。

### 1.1 核心价值主张

- **零摩擦体验**：设置提醒、查找重要邮件、自动化任务、安排会议等功能无缝运行
- **真实个性**：区别于传统 AI 的谄媚语气，具备机智、温暖、人性化的交互风格
- **统一入口**：通过即时通讯平台提供一致的服务体验

## 2. 整体架构设计

Poke 采用 **编排式多智能体系统（Orchestrated Multi-Agent System）** 架构，区别于传统的单线程迭代工具调用模式。

## 3. 核心组件详解

### 3.1 交互代理（Interaction Agent）

交互代理是系统的**唯一用户接口**，承担以下核心职责：

#### 3.1.1 消息路由与编排
- 接收用户请求后，检查现有执行代理名册（Agent Roster）
- 判断是否可复用现有代理或需要生成新代理
- 将多个独立任务并行分发给不同执行代理
- 汇总各代理输出，合成连贯的用户回复

#### 3.1.2 个性化与 UX 管理
- 维护机智、温暖、直接的交互风格
- 明确指令：*"don't act like other sycophantic chatbots"*
- 适应用户的文字风格（大小写、表情符号使用等）

#### 3.1.3 信息过滤
- 评估后台输出是否值得展示给用户
- 对冗余或无关信息调用 `wait` 工具静默丢弃
- 避免用噪音信息干扰用户对话

#### 3.1.4 核心工具
| 工具名 | 功能 |
|--------|------|
| `sendmessageto_agent` | 向执行代理发送消息/生成新代理 |
| `display_draft` | 原样展示邮件/日历事件草稿 |
| `reactto_message` | 对用户消息添加表情反应 |
| `wait` | 静默取消不适当的触发器/通知 |
| `querymedia` | 静默查询媒体信息 |

### 3.2 执行代理（Execution Agents）

每个执行代理是**独立的 LLM 实例**，具有：

#### 3.2.1 核心特性
- **专属系统提示词**：针对特定任务定制
- **独立对话历史**：完整记录请求、工具调用、回复
- **专属工具集**：根据任务需求配置
- **持久化生命周期**：代理不会在任务完成后销毁

#### 3.2.2 工作模式
```
用户请求 → 交互代理分析 → 生成/复用执行代理 → 独立推理循环 → 工具调用 → 结果报告
```

#### 3.2.3 任务所有权
- 代理拥有完整的工作线程（如"Email to Alice"代理负责所有 Alice 相关邮件交互）
- 后续相关请求会路由回同一代理，保持上下文连续性

### 3.3 工具系统（Tools）

工具执行**单一原子操作**，分为以下类别：

#### 3.3.1 Gmail 工具（via Composio）
- 创建草稿
- 发送邮件
- 回复邮件线程
- 转发消息
- 查找联系人

#### 3.3.2 日历工具
- 创建/编辑/删除日历事件
- 事件查询

#### 3.3.3 触发器工具
- 创建/更新/列出自动化提醒和定时任务

#### 3.3.4 浏览器工具
- 完整的浏览器使用能力
- 限制：暂不支持密码等敏感信息输入

### 3.4 任务系统（Tasks）

任务是**复杂操作的编排层**，解决"工具调用膨胀"问题：

#### 3.4.1 邮件搜索任务
- 接收查询后，扇出多个搜索请求（不同时间段、相关关键词）
- 聚合并清理结果
- 防止代理被原始邮件数据淹没

#### 3.4.2 MCP 集成任务
- 定义与 Notion、Linear、Vercel 等外部服务的集成
- 执行代理只需判断相关性，具体 API 调用由任务层处理
- 避免预先暴露所有 API 端点

### 3.5 触发器系统（Triggers）

触发器是 Poke 的**定时任务/提醒系统**：

#### 3.5.1 功能特性
- 支持一次性事件和循环计划
- 自然语言模式：*"every weekday"*、*"first Monday of the month"*
- 作用域隔离：每个触发器归属于创建它的执行代理

#### 3.5.2 技术实现
- 触发器存储在 SQL 数据库
- 后台调度服务每分钟检查到期触发器
- 触发时重新激活原始执行代理执行任务
- 输出路由回交互代理处理

### 3.6 邮件监控（Email Monitor）

#### 3.6.1 工作机制
- 后台 Worker 每分钟获取新邮件
- LLM 分类器评估邮件：紧急程度、所需操作、OTP 代码、发件人重要性
- 超过重要性阈值的邮件发送至交互代理

#### 3.6.2 交互方式
- 交互代理将重要邮件自然融入对话
- 示例：*"By the way, Alice just confirmed lunch."*

## 4. 消息系统设计

### 4.1 消息类型层次

| 标签 | 来源 | 优先级 | 可见性 |
|------|------|--------|--------|
| `<messages>` | 真实用户 | 最高 | 用户可见 |
| `<agent_message>` | 执行代理报告 | 高 | 用户不可见 |
| `<trigger_message>` | 自动化触发 | 中 | 用户不可见 |
| `<email_message>` | 邮件通知 | 中 | 用户不可见 |
| `<interaction_message>` | 开发者通知 | 低 | 用户不可见 |
| `<summary>` | 对话摘要 | 辅助 | 用户不可见 |
| `<context>` | 用户上下文/记忆 | 辅助 | 用户不可见 |

### 4.2 消息处理原则
- **主动操作限制**：仅用户消息可触发操作，其他消息类型需用户确认
- **幻觉维护**：始终表现为单一统一实体，不暴露技术细节
- **静默过滤**：不当触发或无价值更新使用 `wait` 工具静默处理

## 5. 记忆系统（Memory System）

### 5.1 自动记忆管理
系统自动维护用户相关记忆：
- 用户分享的个人信息
- 表达的偏好
- 写作风格和沟通模式
- 历史请求及处理方式
- 重要对话主题

### 5.2 对话摘要机制
- 对话过长时，系统创建摘要（`<summary>` 标签）
- 摘要包含写作风格偏好和已讨论主题
- 自动注入上下文，无需完整历史

### 5.3 交互原则
- **自然融入**：不提及"访问记忆"或"检索信息"
- **智能推断**：对不确定信息进行合理推测，避免让用户重复

## 7. 关键技术点

### 7.1 动态代理生成与持久化

**传统模式**：子代理临时生成 → 执行 → 销毁

**Poke 模式**：
- 代理按需生成后**持久化存储**
- 保留完整的请求、工具调用、回复日志
- 后续相关任务路由回同一代理
- 实现跨对话的上下文连续性

### 7.2 并行任务执行

- 交互代理识别无依赖关系的任务后**并行分发**
- 多个执行代理**同时独立工作**
- 显著提升复杂任务处理效率

### 7.3 MCP（Model Context Protocol）集成

- 支持自定义 MCP 服务器集成
- 解决"工具调用膨胀"问题
- 内置 Notion、Linear、Vercel、Intercom、Sentry 集成
- 用户可上传自定义 MCP 服务器

### 7.4 多源并行搜索

信息查询策略：
- **明确单一来源**：直接使用对应工具
- **多源或不确定**：并行搜索多个数据源
- 优先并行搜索而非猜测最合适来源

### 7.5 用户确认机制

**强制确认场景**：
- 发送/转发/回复邮件
- 创建/更新/删除日历事件

**确认方式**：
- 明确的"是/否"回复
- 正面/负面表情反应（👍 = 确认，👎 = 拒绝）

### 7.6 个性化人格层

| 维度 | 实现 |
|------|------|
| 温暖 | 像朋友一样交流，适度表达关心 |
| 机智 | 微妙幽默和讽刺，不强行搞笑 |
| 简洁 | 无前言后语，不主动提供额外帮助 |
| 适应性 | 匹配用户风格（大小写、表情、长度） |
| 人性化 | 避免企业话术和过度正式语言 |

### 7.7 WhatsApp 政策适配

- 24 小时窗口内可发送自由格式消息
- 窗口外限于预批准模板
- 必要时引导用户切换至 iMessage/SMS

## 10. 架构优势总结

1. **编排式多代理**：区别于单线程迭代，实现真正的并行处理
2. **代理持久化**：跨对话上下文保持，提升任务连续性
3. **任务所有权**：每个工作线程由专属代理管理，职责清晰
4. **智能过滤**：后台噪音不干扰用户对话
5. **自然人格**：突破传统 AI 助手的机械感
6. **可扩展集成**：MCP 协议支持自定义服务接入
7. **多源并行搜索**：高效获取分散在不同服务的信息
