## 问题根因
- 结构化输出解析失败：当启用 JSON schema 时，解析 `function_call.arguments` 或 `message.content` 失败，`resp=None`，后续代理下标访问触发异常（日志见 qiaoyun_runner.log:230-299）。
- SDK字段差异：基类仅兼容 `message.function_call`，未兼容 `message.tool_calls`（Ark/Doubao），导致未取到结构化内容（framework/agent/llmagent/base_singleroundllmagent.py:195-208）。
- 模板对空对象下标访问：模板格式化依赖的上下文字段为 None 或缺失，`format(**context)` 抛异常（base_singleroundllmagent.py:121-130）。
- 主链路对空响应未兜底：`QiaoyunChatAgent` 直接访问 `resp["MultiModalResponses"]`，失败后整个链路不生成 `outputmessages`（qiaoyun/agent/qiaoyun_chat_agent.py:59-62）。

## 修复方案
### 1) 基类增强：统一结构化解析与兜底
- 兼容 SDK 字段：在 `framework/agent/llmagent/base_singleroundllmagent.py`：
  - 优先读取 `response.choices[0].message.tool_calls[0].function.arguments`（若存在），其次 `message.function_call.arguments`，最后 `message.content`。
  - 当使用 Ark/Doubao 时，支持 `tool_calls` 数组；当无工具调用时，启用 `response_format` 或强指令要求纯 JSON。
- 失败兜底：
  - 解析失败时，返回符合 schema 的默认字典（字段置为“空”、空数组或空字符串），而不是抛异常；同时 `status` 设置为 `SUCCESS`，避免“FINISHED(失败)”误导主链路。
  - 在 `_prehandle` 前使用 `default_input` 填充 context，防止模板下标访问 None。
- 日志与调试：记录 `model`、`tool_calls/function_call` 检测结果、失败时的原始 `content` 截断片段，便于回溯。

### 2) 代理级默认值与健壮性
- 给各代理设置 `default_input`：
  - `QiaoyunQueryRewriteAgent`、`QiaoyunContextRetrieveAgent`、`QiaoyunChatResponseAgent`、`QiaoyunPostAnalyzeAgent` 在构造时传入必须字段的默认空结构，确保模板安全。
- 响应字段检查：
  - `QiaoyunChatAgent` 在读取 `result["resp"]["MultiModalResponses"]` 前检查 `resp` 及键存在；若缺失，构造 `MultiModalResponses=[{"type":"text","content":"我现在网有点卡，晚点回你哈"}]`。

### 3) 处理器兜底回复
- 在 `qiaoyun/runner/qiaoyun_handler.py:136-171` 的主循环中：
  - 若所有子代理未产出 `AgentStatus.MESSAGE`，或 `resp_messages` 为空，使用 `send_message_via_context(...)` 发送兜底文本，保证用户端“有回声”。
  - 同步将错误信息写入 `conversation_info`，便于后续排查。

### 4) JSON模式强制（可选增强）
- 对使用 Ark/Doubao 的代理，通过 `extra_args` 注入 `response_format={"type":"json_object"}` 或明确 tool 选择，减少非 JSON 返回。
- 在 prompts 中加入简短示例 JSON 以及“仅返回纯 JSON”的强调，降低跑偏。

## 影响与风险
- 解析兜底会让部分复杂能力退化为“简短文本回复”，但优先保障用户体验与链路可用性。
- SDK 字段兼容需谨慎测试不同模型与客户端的返回结构，避免误判解析路径。
